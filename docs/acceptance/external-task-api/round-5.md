# round-5（2026-07-17）：「复制」语义改为复制原密钥明文 · 全绿

用户澄清 round-4 的「复制」：要的是**复制原密钥**，不是按配置克隆生成新钥。

## 语义与取舍

- 明文随密钥条目落盘 `runtime/api-keys.json`（`就存一下明文密钥`）：发起端配置（scrumws-ingest.json）本就在同机存明文，服务端留存不扩大风险面；管理面只听 127.0.0.1。鉴权比对仍走 sha256。
- 列表直接外带 `plaintext`（不再单设 reveal 端点——本机管理面同一信任域，不过度设计）；hash 仍不外带。
- UI「复制」= 原文进剪贴板（成功按钮闪「已复制」；剪贴板不可用降级弹窗展示手动复制）；明文留存前的旧版钥 `plaintext=null` → 按钮置灰。
- 创建横幅「明文只显示这一次」文案随之废除（改「新密钥」，随时可从列表复制）。

## 证据（r8 隔离实例）

- **API 38/38 PASS**：新增 R1（列表明文与创建返回一致；注入的旧格式钥 plaintext=null）；A6 改为「列表不含 hash」（明文外带是设计内）。
- **UI 8/8 PASS**：U7 改为点「复制」→ 剪贴板路径实测走通（headless 亦过）。
- **生产两把钥明文已回填**（来源 baibu scrumws-ingest.json；逐把 sha256 校验匹配才写，写后读回防心跳并发覆盖）——PR 合并 + 桌面端重启后「复制」立即可用，无旧钥置灰问题。
