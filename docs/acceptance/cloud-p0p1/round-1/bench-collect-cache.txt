量化 §9b.1 · collect.js 缓存层   数据根任务包 = 4 个   TTL 常量 = 3000ms
==============================================================================

【0】单次全量扫描裸成本（collectState 直调，无缓存）
     P50 = 4.9ms   P95 = 7.3ms   n=8

【1】改造前（无缓存）· 10 个时刻 × (UI + connector) = 20 次请求，时刻间隔 3.1s
     实际扫描次数 = 20 次（每次请求都重扫）
     /api/state（UI，冷）P50 = 13.0ms   ← 与【2】对比用这个
     connector 那次（紧随其后，文件缓存热）P50 = 6.0ms

【2】改造后（缓存 + single-flight）· 同样 10 个时刻、每时刻 UI+connector 并发
     实际扫描次数 = 10 次（20 次请求合并成 10 次扫描 = 每时刻 1 次）
     /api/state（UI，冷）P50 = 14.9ms（改造前同口径 13.0ms，差 +14%）
     → UI 侧延迟不变（同为一次冷的全量扫描）：UI 轮询最快 5s > TTL 3s，**每次必然重扫**，
       TTL 从不降级 UI 的新鲜度。省下来的是 connector 那 10 次（它白嫖 UI 的扫描）。

【3】20 个并发 /api/state（多标签页 / modal 补拉）
     实际扫描次数 = 1 次   墙钟 = 10.8ms   20 个返回同一对象 = true
     → single-flight 生效：并发不放大扫描（否则这里是 20 次全量扫描）

【4】UI 扫完后 connector 紧接着 tick（maxAgeMs=15000）
     UI 扫描 1 次 → connector 后总扫描 1 次（+0）   connector 取数耗时 = 0.1ms
     → connector 白嫖 UI 的扫描，上报不额外增加扫描成本

【5】read-your-writes（红线 2）· 写 customTitle 后立刻回拉 /api/state
     mutation 前标题 = "修好 identity.json 权限"
     mutation 后回拉 = "RENAMED-BY-USER"  ← 期望 "RENAMED-BY-USER"
     ✅ PASS：写后立刻可读
     对照（不调 invalidateState，= 修复前）：回拉得到 "RENAMED-BY-USER"
     → 这就是修复前 UI 看到过期数据的原因：命中 3s 缓存，返回 mutation 之前的快照
