# round-3 · §9b.1 在真实数据根规模上复测 collect 缓存

**结论：round-1 的外推模型低估 4.4 倍，已纠正；但缓存层的设计结论不变，且证据更强。**

日期：2026-07-17　脚本：`scripts/bench-real-dataroot.mjs`

---

## 1. 为什么要复测

round-1 的 bench（`round-1/bench-collect-cache.txt`）跑在沙箱上：**4 个任务包、0 张 CLI 观察态卡**，得出 `collectState()` P50 = 4.9ms，并外推「~0.5ms/包线性长，300 包 = 152ms」。

那个外推**漏了成本大头**：CLI 观察态卡要**反读 Claude Code 自己的 jsonl** 数后台子 agent（`collect-cli.js:195`），而沙箱里一张 CLI 卡都没有——这块成本压根没进模型。设计 §9b.1 当时就标注了这是「P1 唯一的技术不确定点」，本轮兑现。

## 2. 测量方法（为什么不在真实数据根上直接跑）

`collectState()` **不是纯只读**：`collect-cli.js:57-58` 会 `unlinkSync` 掉 pid 已死的 sentinel。用户的 app 正跑着（8799），不该在它的真实 `~/.scrumws` 上做写操作。

故建**镜像沙箱**：把真实数据根的 `runner-state` / `runner-archive` / `cli-watchlist.json` / `runner-config.json` 拷到 `.sandbox/real-mirror/`（共 378K）。数据量与真实完全一致，且 `cli-watchlist.json` 里的 `jsonlPath` 仍指向真实的 `~/.claude/projects`（**1683 个 jsonl / 564MB**）——反读那部分是只读的，正是要测的成本。脚本首行即拒绝在路径以 `.scrumws` 结尾的数据根上运行。

## 3. 数据规模对照

| | round-1 沙箱 | 真实规模（本轮） |
|---|---|---|
| 活跃任务包 | 4 | 17 |
| 归档任务包 | 0 | 9 |
| **CLI 观察态卡** | **0** | **25** |
| 卡片总数 | 4 | **51** |
| CC jsonl | 无 | 1683 文件 / 564MB |

分桶实测：`{"plan":0,"processing":1,"queued":0,"done":11,"awaitingHuman":19,"archived":20}`

## 4. 实测结果

### 【1】`collectState()` 裸成本

```
P50 = 112.2ms   P95 = 126.7ms   min = 106.6ms   n=10
对照 round-1 沙箱（4 包 / 0 CLI 卡）：P50 = 4.9ms
摊到每张卡：2.2ms/卡   对照 round-1 外推模型：~0.5ms/包
```

**真实规模是沙箱的 23 倍；每卡成本是模型的 4.4 倍。**

→ **round-1 的「300 包 = 152ms」是错的，按实测应为 ~660ms。** 该结论已在 `round-1.md` 就地标注纠正。

### 【2】single-flight（20 并发）

```
实际扫描 = 1 次   墙钟 = 109.6ms   20 个返回同一对象 = true
若无 single-flight：20 × 112ms ≈ 2.2s 连续扫描
```

### 【3】写代次 / read-your-writes

```
填充后扫描数 0 → TTL 内再取 0（+0 = 命中缓存）
invalidate 后再取 1（+1 = 强制重扫）
✔ 缓存生效且写后必新鲜
```

### 【4】connector 白嫖

```
UI 扫描 1 次 → connector 后总扫描 1 次（+0）   connector 取数耗时 0.0ms
```

## 5. 结论：模型错了，设计对了

**低估被纠正，但每一条设计结论都站得住，而且论据更硬：**

| 设计决定 | 沙箱数据下的说服力 | 真实数据下的说服力 |
|---|---|---|
| **single-flight** | 20 并发省掉 19×4.9ms ≈ 93ms，锦上添花 | 20 并发省掉 **2.2 秒**连续扫描，**承重墙** |
| **connector 不得触发额外扫描** | 省 4.9ms/15s，几乎无所谓 | 省 **112ms/15s**，实测 **+0 次扫描** |
| **写代次而非纯 TTL** | 17ms 回拉命中新值 | 真实规模下同样成立（+0 命中 / invalidate 后 +1） |

**TTL 从不降级 UI 新鲜度**这条也不变：UI 轮询最快 5s > TTL 3s，每次必然重扫。

## 6. 新增的已知事实（非 bug，但要知道）

1. **`/api/state` 在真实规模上是 112ms 量级**。UI 每 5s 轮一次 ≈ 2.2% 的 CPU 占空比，可接受。
2. **成本主要由 CLI 观察态卡驱动，不是任务包**。26 个任务包 + 25 张 CLI 卡 = 112ms；沙箱 4 个纯任务包 = 4.9ms（≈1.2ms/包）。据此推算 CLI 卡约 3ms/张，是任务包的 ~2.5 倍——因为它要反读整个 jsonl。**未单独测量该分解，属推算，不作为结论。**
3. **容量规划按 ~2.2ms/卡**（实测摊薄值），不要用 round-1 的 0.5ms/包。CLI 卡越多越贵。

## 7. 未收敛

- CLI 卡 vs 任务包的成本分解是**推算不是实测**（§6.2）。要精确的话得分别构造两种卡片做对照，本轮没做。
- 未测 jsonl 单文件体积对反读成本的影响（564MB 分布在 1683 个文件，单文件均值 ~344KB；超大单文件会不会劣化未知）。
